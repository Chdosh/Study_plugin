# Study Agent Development Instructions

## 1. 项目身份

本项目是一个本地优先的 Windows 桌面 AI 学习系统，目标是逐步成为长期个人学习教师。

当前工作必须服务于可恢复的学习闭环：目标澄清、分层计划、今日重点任务、学习执行、过程记录、结果提交、评估、复盘与后续调整。不要把长期愿景当作当前 MVP 的硬性范围。

## 2. 事实源体系

本项目不使用单一线性优先级。遇到冲突时，先判断冲突属于哪类事实，再读取对应事实源。

用户当前显式请求优先于仓库内规则。Product Truth 用于补充用户未明确说明的产品行为；如果用户当前请求明确要求改变 Product Truth，先按请求执行，并同步更新对应文档。

| 事实类型 | 最高事实源 | 辅助事实源 |
| --- | --- | --- |
| 产品定位、核心概念、当前产品行为、功能边界 | `docs/PRODUCT_TRUTH.md` | `docs/MVP_SPEC.md` |
| 已落地技术选择、依赖、脚本、构建方式 | `package.json`、lockfile、配置文件、当前代码 | `docs/ARCHITECTURE.md` |
| 数据库结构和持久化事实 | Drizzle schema、迁移文件、当前数据库访问代码 | `docs/TESTING_AND_MIGRATIONS.md`、`docs/AI_AND_DATA_RULES.md` |
| AI 上下文、结构化输出、prompt、模型调用边界 | 当前 AI 相关代码与测试 | `docs/AI_AND_DATA_RULES.md` |
| Electron 安全、隐私、权限和敏感数据边界 | `docs/SECURITY_AND_PRIVACY.md` | preload、IPC、main/renderer 当前实现 |
| UI 语言、交互、视觉和页面职责 | `docs/UI_GUIDELINES.md` | 当前 renderer 实现、用户提供的设计要求 |
| 正式 Task-ID 工作流、证据和交付协议 | `.agent/WORKFLOW.md` | `.agent/TASK.md`、`.agent/STATUS.json` |
| 新对话短交接、近期风险和下一步 | `docs/PROJECT_MEMORY.md` | 相关代码和专题文档 |

如果文档与代码冲突，不要默认认为当前实现正确。先按上表定位事实类型：产品行为以 Product Truth 为准，技术落地以当前代码和配置为准，安全和数据边界以对应专题文档与实现共同校验。无法判断时，在回答中说明冲突和采用依据；必要时先请求用户确认。

## 3. 必读与按需读取

每次实现或 review 前必须读取：

1. 本文件。
2. 与当前任务直接相关的源文件、组件、服务、类型和测试。
3. 当前代码不足以判断、任务涉及对应规则，或代码与文档可能冲突时，读取相关专题文档。

`docs/PROJECT_MEMORY.md` 不是每次任务的必读文件。仅在需要了解近期未解决风险、接续长期任务、处理跨模块历史决策或用户明确要求时读取。

需要补充事实时读取：

| 任务类型 | 读取文档 |
| --- | --- |
| 产品流程、范围、MVP、规划体验、信息架构、功能取舍 | `docs/PRODUCT_TRUTH.md`，必要时再读 `docs/MVP_SPEC.md` |
| 架构、IPC、preload、main/service、依赖、技术选型 | `docs/ARCHITECTURE.md` |
| AI、prompt、数据模型、计划版本、上下文、RAG、知识库 | `docs/AI_AND_DATA_RULES.md` |
| Electron 权限、监控、密钥、隐私、敏感数据 | `docs/SECURITY_AND_PRIVACY.md` |
| UI、交互、视觉重构、页面状态 | `docs/UI_GUIDELINES.md` |
| 测试、验证、数据库 schema、迁移 | `docs/TESTING_AND_MIGRATIONS.md` |
| 参考项目调研 | `docs/REFERENCES.md` |
| 使用 Task-ID 或 `.agent/TASK.md` 的正式任务 | `.agent/WORKFLOW.md` |

不要默认读取全部文档；普通局部 Bug 修复不因关键词命中自动扩展阅读范围。不要把历史归档、旧报告或示例当作当前规范。`docs/Example.md` 只能作为规划输出参考样例，不得成为模型或实现的隐式硬依赖。

## 4. 当前技术边界

当前技术栈以已落地代码和配置为准：

* Electron + React + TypeScript
* SQLite/libSQL + Drizzle ORM
* OpenAI-compatible DeepSeek client
* Zod runtime validation
* typed preload API + narrow IPC
* Vitest

未经用户明确批准，不迁移框架、样式系统、数据库、状态管理、AI 客户端或主要依赖。

## 5. 语言要求

开发记录、设计理由、迭代说明、迁移上下文和项目记忆更新使用中文。

用户可见应用 UI 使用中文。代码标识符、数据库列、API 字段、TypeScript 类型和必要技术名词可以使用英文，以保证可维护性。

避免中英文混杂的 UI 标签，除非英文术语必要或已被广泛使用。

## 6. 通用开发规则

修改前：

1. 先检查当前实现和相关测试。
2. 识别最小相关变更范围。
3. 判断是否影响产品行为、数据、IPC、AI 行为、安全边界或既有用户流程。
4. 对需求含糊处说明假设；优先采用最小、可逆且不改变产品含义的假设继续执行。只有涉及删除用户数据、数据库破坏性迁移、公开接口破坏或无法判断的产品方向时，才请求用户确认。

实现中：

* 做能完成目标的最小一致变更。
* 一次只处理一个页面、模块或端到端流程。
* 复用现有组件、服务、类型、测试和项目模式。
* 不做无关重构，不无必要地批量重命名文件。
* 不新增依赖，除非现有技术无法合理完成任务。
* 不创建看似完整但没有真实数据流的占位功能。
* 优先显式、可维护的代码，而不是聪明抽象。
* UI-only 任务不得修改数据库结构或业务状态机。
* 视觉重构不得改变业务逻辑，除非用户明确要求。
* 可以删除或收敛已确认冗余的内部实现，但必须先确认它不承载 Product Truth 要求的能力，并保留必要的数据兼容或迁移路径。
* 不以“向后兼容”为理由保留已确认错误或冗余的内部行为；面向用户数据和公开接口的兼容仍需谨慎处理。

大型或高风险变更不强制先新增文档。优先在当前任务中给出简短设计说明、影响范围和验证方案；只有用户要求、Task-ID 工作流要求，或变更需要长期留痕时，才新增或更新正式文档。

## 7. 最高级禁令

Git 与工作树：

* 禁止自动提交。
* 禁止运行 `git reset --hard`、`git clean`、`git checkout --`、`git restore`、`git stash`、`git push`。
* 不得回滚、覆盖或混入用户已有变更。发现与当前任务无关的用户修改时保留并绕开；只有修改同一文件的同一区域、继续操作可能覆盖用户内容时，才停止并要求人工决策。

数据：

* SQLite 是 durable source of truth。
* schema 变更必须使用迁移，不得只靠启动时 ad hoc SQL 改生产结构。
* 未经用户明确批准，不删除表、列或用户数据。
* AI 输出是 proposal，必须经过运行时校验，并在必要时经用户确认后才能落库或影响计划。

AI：

* 不得把完整用户历史发送给模型。
* 不得因为真实模型单次失败就放宽业务安全边界、编造必填字段或让 AI 直接改变状态。
* 结构化 AI 输出必须做运行时校验，失败时不得静默补造关键字段。
* 真实 DeepSeek 合约测试是 opt-in，不作为默认自动 PASS 门槛。

安全与隐私：

* Renderer 不得直接访问 SQLite、Node.js 文件系统、操作系统监控 API、API key、Electron `safeStorage` 或无限制 IPC。
* 不得明文存储 API key，不得在日志、错误、AI 调用记录或数据库快照中泄露 secret。
* 不得自动执行 AI 生成的代码或 shell 命令。
* 不得实现截图、录屏、键盘记录、剪贴板监听、麦克风/摄像头采集、完整浏览器历史收集、私信内容收集、强制锁屏或隐藏后台监控。

## 8. 验证标准

验证范围应与变更风险匹配，不是所有变更都需要跑全部检查。

| 变更类型 | 必做 | 可选/按风险 |
|----------|------|------------|
| 纯文档/Agent 规则/注释 | diff 审阅、关键词扫描 | 不跑代码检查 |
| CSS/样式调整 | typecheck | build（可选） |
| 组件 JSX/布局（不涉及数据流） | typecheck | build、浏览器预览 |
| 组件 JSX/布局（涉及数据流） | typecheck、相关测试 | build |
| 服务/状态机/业务逻辑 | typecheck、相关测试 | build |
| IPC/preload 变更 | typecheck、相关测试、build | |
| 数据库/schema/迁移 | schema 检查、迁移测试 | typecheck |
| AI/prompt/schema 变更 | 结构化 schema 测试、失败路径测试 | 完整单元测试 |
| 重命名/重构（不变行为） | typecheck | 相关测试 |
| 删除文件 | typecheck（检查遗留 import） | |
| 依赖变更/配置脚本 | typecheck、build | 测试 |
| 多模块跨层变更 | typecheck、全部测试、build | |

规则说明：
* 不得声称未运行的测试已经通过。命令失败时必须说明命令、失败现象、可能原因，以及是否由本次变更引入。
* 不得以 PROJECT_MEMORY.md 中历史验证记录为模式自动执行不存在的脚本（如已删除的 `scripts/electron-gui-smoke.mjs`）。验证命令以当前 `package.json` scripts 和实际存在的文件为准。
* 真实 DeepSeek 合约测试仅在用户要求或明确 opt-in 时运行。
* 正式 Task-ID 任务的证据记录、命令日志和交付协议见 `.agent/WORKFLOW.md`。普通小型交互任务不强制生成完整证据目录。

## 9. 完成标准

开发任务完成时必须满足：

* 请求行为已实现，或已明确说明未完成原因。
* 相关 UI 状态、数据持久化、AI 校验和用户确认规则按任务需要处理。
* 已执行相关测试或手工检查。
* 未有意改变无关功能。
* 需要长期留痕时，已更新对应专题文档或 `docs/PROJECT_MEMORY.md`。

最终回复使用中文，简明报告完成内容、修改文件、验证方式和结果、未解决问题或风险。不要为每个小任务强制输出固定六类报告；不要把推测或未验证工作报告为完成。

## 10. 项目记忆

`docs/PROJECT_MEMORY.md` 是新对话短交接入口，不是完整日志，也不是产品事实源。

只在完成有意义的开发任务、架构决策、schema 变化、产品流程变化或发现需要后续接手的技术风险后更新项目记忆。轻微 CSS 调整、临时调试、纯格式修正或本文件这类治理性收敛，不默认更新项目记忆。

项目记忆更新使用中文，保持短小，优先记录日期、本次完成、关键决策、修改范围、验证结果、尚未解决和推荐下一步。不要粘贴大段代码、完整日志或临时推理。
