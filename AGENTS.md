# Study Plugin Agent Rules

状态：CURRENT
生效日期：2026-07-13

## 1. 项目方向

这是一个本地优先的 Windows 桌面 AI 学习教师，不是聊天壳、待办列表或单纯计时器。

所有工作最终服务于可恢复的学习闭环：

```text
了解目标与约束
→ 建立分层学习路径
→ 生成当前可执行内容
→ 执行、提问、暂停和恢复
→ 保存结果并评价
→ 记录薄弱点与经验
→ 复盘并调整后续安排
→ 持续循环直到目标完成
```

产品可以分阶段实现，但不能通过制造第二套状态、假交互或不可恢复流程换取表面完整。

## 2. 事实优先级

1. 用户当前对话中的明确要求。
2. 本文件中的长期协作约束。
3. 当前代码、schema、迁移、配置和测试，用于确认“现在实现了什么”。
4. 用户在当前任务中明确指定的新规格。

当前代码和测试都可能包含 Bug，不能自动证明产品行为正确。

`docs/_legacy/**`、`docs/archive/**`、`.lavish/**`、`.superdesign/**` 及旧 `.agent`、`.opencode` 工作流没有规范效力。除非用户明确要求历史追溯或兼容核对，否则不要读取。旧规则必须同时与当前代码、当前方向和真实流程一致，改写进当前规则后才重新生效。

## 3. 每次任务都必须遵守

### 3.1 一个业务事实只有一个正式来源

目标、Current Guide、当前 Task/Action、活动 Session、提交和评价状态都只能有一个正式来源。Renderer 不得维护第二套业务位置，不同页面不得各自推导同一状态。

### 3.2 AI 生成内容，程序推进状态

AI 输出必须经过运行时和业务校验。AI 不得直接标记完成、改变 Current Context、覆盖历史或未经确认永久修改计划与用户画像。

### 3.3 原始事实和用户数据优先

用户原话、提交、实际 Session、当时生效的计划、评价和用户确认属于原始事实。摘要、复盘、画像和知识推断属于派生信息，不能覆盖原始事实。

用户提交必须先持久化再调用 AI。AI 失败、应用关闭或后续写入失败时，提交仍需保留并可重试。

### 3.4 用户保有控制权

重大计划调整、Current Guide 冲突、删除数据和不可逆操作必须由用户决定。自动修复只处理可唯一推导的状态。

### 3.5 使用纵向业务切片

“最小变更”指最小可完整验收的业务切片，不是最少文件或单个页面。真实症状跨越 Renderer、IPC、Module、Store 和数据库时，必须跨层修复，但不得顺手重构无关区域。

每个业务修改先明确：入口状态、用户动作、业务 owner、数据变化、成功终态、失败恢复和重复执行语义。

### 3.6 不制造表面功能

禁止新增无真实回调、无真实数据流、硬编码假计时、伪统计、永远禁用的按钮或只为截图存在的生产功能。

## 4. 业务闭环与界面必须整体理解

以下内容不能按页面拆开理解，因为它们共同描述同一条学习闭环。

### 4.1 当前业务语义

* 学习目标连接长期结果和全部历史。
* Roadmap Stage 表示阶段方向；Short Plan 表示近期粗粒度安排；Daily Guide 表示当前学习单元执行稿。
* Task 是正式成果、提交和评价的中心单位。
* Action 是 Task 内的执行步骤，本地完成或跳过，不单独触发 AI 评价。
* Focus Session 是一次实际执行记录，可以暂停和恢复，不等于 Task 完成。
* 一个 Task 可以跨多个 Session；全部 Action 终态后进入“等待提交”，评价通过后 Task 才进入“已完成”。

所有页面和命令必须共用以下状态语义：

```text
未开始 / 执行中 / 已暂停
等待提交 / 评价中 / 等待重新评价 / 等待修改
已完成 / 已跳过 / 已暂缓
```

### 4.2 当前信息架构

主导航基线为：

```text
概览 / 学习 / 记录 / 设置
```

* 概览回答“我朝什么目标前进，下一步关注什么”，只展示目标、阶段、学习路径、当前重点摘要和待处理事项，不复制执行详情、实时计时和事件记录。
* 学习回答“我现在具体做什么”，承载当前 Task、Action、Session、提问、提交和评价恢复。
* 记录回答“过去发生了什么，结果如何，后来怎样调整”，统一收纳 Session、Action、提交、评价、问题、知识、复盘和计划版本，不提供学习控制。
* 设置只负责模型、偏好、隐私、本地数据和诊断配置。
* 永久壳层只保留必要导航和全局状态；AI 导师只在学习上下文中出现。

改变信息架构时必须整体审查页面职责和状态流，不能只改导航文字或单个页面。

### 4.3 UI 必须服从业务状态

* 每个页面状态只有一个视觉主操作，且对应用户此刻最重要的下一步。
* 计划、执行和历史必须分层，同一完整内容不得跨页面重复。
* 全部 Action 终态后立即显示提交状态，不得继续出现越界步骤序号或完成/跳过操作。
* 不显示数据库 ID、内部字段、英文诊断、伪计时、伪百分比、空统计或开发占位。
* 错误必须说明数据是否保留和用户下一步，并提供真实可执行的恢复入口。
* 导航、按钮、标签和抽屉支持键盘；不能只依赖颜色表达状态。
* 操作栏不能覆盖正文，响应式变化不能重置输入、Session 或当前位置。
* 视觉重构不能删除真实业务能力，也不能新增只为截图存在的功能。

涉及业务状态的 UI 修改，先列出：

```text
状态 | 页面摘要 | 主操作 | 次级操作 | 禁止操作 | 错误恢复
```

Renderer 必须消费共享业务状态或派生状态，不得用页面局部条件重新组合第二套状态机。

## 5. 仅按需读取的实现参考

只有任务涉及架构、IPC、Module、Store、数据库、迁移、AI 上下文、安全边界、旧表或旧命名时，才完整读取：

`docs/rules/ARCHITECTURE_DATA_COMPATIBILITY.md`

普通 UI、文案、局部组件和只读审查不需要默认加载该文件。不要继续读取无关归档。

## 6. 基本工作流程

修改或 review 前：

1. 运行 `git status --short`，识别用户已有变更。
2. 读取当前任务直接相关的代码、类型和测试。
3. 确定正式业务入口和持久化事实源。
4. 判断任务是否影响产品、数据、AI、IPC、安全或恢复流程。
5. 对 Bug 先复现；对 UI 流程先列状态和主操作。

用户要求审查、解释或诊断时默认只读，不直接修改。用户要求修复或实现时，完成请求所需的最小完整切片并验证到用户终态。

## 7. 诊断与完整流程验证

Bug 任务必须先建立能捕获真实症状的反馈环，优先使用真实应用操作、跨层集成测试或真实持久化状态。先复现，再提出 3–5 个可证伪假设，然后逐项验证，不能先修改再寻找解释。

修复时应将真实复现固化成红色测试或等价检查。测试 fixture 必须能通过真实调用链到达，禁止构造业务上不可能出现的状态取得假绿。

涉及主学习闭环时，至少验证：

```text
创建或恢复目标
→ 获得当前学习内容
→ 开始 Session
→ 完成或跳过 Action
→ 提问并返回主线
→ 提交结果
→ 评价成功或失败重试
→ Task 与 Session 正确收口
→ 下一任务或记录/复盘
→ 重启后恢复到一致位置
```

修改相关分支时，还需覆盖暂停、跳过、评价超时、重复提交、应用中断和多个可恢复 Guide。

验证按风险匹配：

* 文档：diff 审阅和失效引用扫描。
* 纯 CSS：typecheck 和真实窗口检查。
* JSX/交互：typecheck、相关测试和真实 Electron 状态检查。
* 状态机、service、store：逻辑测试、跨层集成测试和完整用户路径。
* IPC/preload：typecheck、相关测试和 build。
* schema/迁移：新旧数据库迁移、备份和恢复路径。
* AI schema/prompt：正常、非法、超时和失败恢复。
* 多层闭环：typecheck、全部测试、build 和真实 Electron 主路径。

## 8. 技术与安全底线

当前技术栈以 `package.json`、lockfile 和代码为准。未经用户明确批准，不迁移框架、数据库、状态管理、AI 客户端或主要依赖。

必须遵守：

* Renderer 不直接访问 SQLite、Node 文件系统、API key、safeStorage 或操作系统监控 API。
* SQLite 是 durable source of truth；schema 变化必须使用迁移。
* 未经用户明确批准，不删除表、列或用户数据。
* 不在日志、错误、AI 调用或数据库快照中泄露 secret。
* 不自动执行 AI 生成的代码或 shell 命令。
* 不实现截图、录屏、键盘记录、剪贴板监听、麦克风/摄像头采集、完整浏览历史收集、私信收集、强制锁屏或隐藏监控。
* 真实 DeepSeek 测试是 opt-in，除非用户明确要求，不自动消耗额度。

## 9. 工作树和 Git

* 用户已有修改必须保留；不得回滚、覆盖或混入无关改动。
* 只有同一文件同一区域无法安全合并时才停止并请求处理。
* 禁止自动提交。
* 禁止运行 `git reset --hard`、`git clean`、`git checkout --`、`git restore`、`git stash` 或 `git push`。
* 不做无关重命名、批量格式化或依赖升级。

## 10. 文档规则

* 本文件是唯一每次默认读取的仓库规则。
* `docs/README.md` 只提供文档索引。
* 低频实现参考按任务读取，不与本文件重复完整产品闭环。
* 新增长期文档必须写明状态、日期、适用范围和失效条件，并登记到 `docs/README.md`。
* 临时推理、修复日志和历史方案只能进入 `docs/_legacy/`，不能继续堆入常驻规则。

## 11. 完成标准

完成时必须满足：

* 用户请求的真实行为已实现，或明确说明阻塞原因。
* 修改覆盖最小完整业务切片，没有只修表面症状。
* 页面、状态、持久化和恢复语义一致。
* 已执行与风险匹配的测试或真实检查。
* 未覆盖用户已有修改，未改变无关功能。
* 没有留下假交互、调试日志、临时脚本或第二套事实源。

最终回复使用中文，简明说明修改内容、关键文件、验证结果、真实数据影响和剩余风险。不得把未运行的检查报告为通过。
