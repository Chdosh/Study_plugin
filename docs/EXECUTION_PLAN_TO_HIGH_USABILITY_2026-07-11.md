## 2. 目标版本定义

### M1：内部稳定版

主链在测试数据库和旧数据库上可靠运行，不出现错误自动推进、无法恢复或迁移噪声。适合开发者自己每天使用。

### M2：真实试用版

普通用户无需理解内部状态机即可完成目标创建、学习、提问、提交、评价、暂停恢复和下一单元推进。适合小范围真实用户连续试用。

### M3：高可用版

连续使用 14 天、累计 30 次以上 Session，不丢数据、不错误推进、不需要手工修库；关键变化可解释、可撤销、可恢复。

## 3. 固定执行顺序

```text
P0 工作树稳定与事实同步
→ P1 数据迁移与启动一致性
→ P2 提交评价事务闭环
→ P3 计划变更和上下文适配
→ P4 UI 交互基线
→ P5 分支、历史和知识体验
→ P6 兼容、备份和端到端验收
→ P7 真实用户连续使用
```

不得同时全面重构数据库、状态机和 UI。每个阶段通过验收后才进入下一阶段。

### P1.1 migration 噪声和事实源收敛

对应：Q2。预计贡献：+3%。

### P1.2 R2 显式生成失败状态

预计贡献：+2%。

状态：`completed`

2. AI schema、超时、网络失败分别记录错误类别。

### P1.3 R4 启动一致性审计

预计贡献：+4%。

状态：`completed`

3. 多解或冲突状态不得自动选择，返回结构化冲突。
4. UI 提供“恢复推荐位置”“查看冲突”“保留数据并稍后处理”。
5. 覆盖应用在 Action、Submission、Evaluation、Guide 关闭过程中退出的恢复测试。

验收：代表性中断状态要么安全恢复，要么明确阻断；不得静默推进或删除数据。

### P2 T1 提交—评价—推进事务闭环

预计贡献：+5%。

任务：

1. 将 Evaluation、Decision、Task 状态、Runtime Pointer 和 Focus Session 完成收敛到可恢复事务。
2. 定义 `waiting / evaluating / failed / applied` 的持久状态。
3. 评价成功但状态推进中断时可幂等重放，不再次调用 AI。
4. 同一 Submission 最多产生一个有效 Evaluation 应用结果。
5. 将 AI 错误、校验错误、数据库错误映射为统一结构化错误。

验收：在每个写入节点模拟退出，重启后都能恢复；任务不会重复完成；下一任务不会被跳过。

### P3.1 T2/T3 计划 Proposal 与版本

预计贡献：+4%。

任务：

1. 建立 PlanAdjustmentProposal 的读取、确认、拒绝和过期流程。
2. AI 只生成 proposal，不直接更新 ShortPlanDay。
3. 接受时保存 before/after、原因、来源 Evaluation/Review 和影响范围。
4. 锁定已执行、已激活和用户明确锁定的学习单元。
5. Review 展示计划变化差异与撤销边界。

验收：拒绝不改正式计划；重复确认幂等；已执行内容不可覆盖；计划历史可追溯。

### P3.2 C1/C2 Learner Context 与冲突仲裁

预计贡献：+5%。

任务：

1. 完成所有核心 AI operation 的允许、必需和禁止字段矩阵。
2. 建立 LearnerFact proposal：如 Windows、DeepSeek、学习基础、讲解偏好。
3. 区分单次任务要求、当前目标偏好和长期稳定事实。
4. 使用确定性优先级处理冲突：用户明确确认 > 实际提交证据 > 连续评价 > 旧事实 > 单次 AI 推断。
5. 无法仲裁时保留冲突并请求用户确认，不静默覆盖。
6. 稳定事实只影响相关未来操作，不默认重写历史计划。

验收：用户说明使用 Windows/DeepSeek 后，相关后续教程自动采用该环境；一次临时要求不会污染全局画像；事实可查看、修改和撤销。

### P3.3 Learning Branch 统一

预计贡献：+3%。

任务：

1. 用统一 Branch 承载提问、排错、额外练习和概念探索。
2. Branch 必须绑定 Goal、Task、Action 和打开时的运行位置。
3. 关闭时可选择：仅关闭、沉淀知识、提议 LearnerFact、提升为正式 Task。
4. Branch 不改变主任务进度；关闭后恢复原位置。

验收：额外练习可以独立持续，不影响主任务完成度；提升为正式任务必须用户确认。

完成 P2/P3 后，核心长期适应能力形成，综合成熟度目标约 78%～82%。

### P4 UI 信息架构与交互基线

预计贡献：+7%。

本阶段才进入系统性 UI 重构。先做交互和信息层级，再做视觉润色。

任务：

1. 建立全局 CommandPolicy：不可操作按钮禁用并说明原因。
2. 将顶部全局提示改为操作附近的局部反馈；保留全局错误摘要。
3. Today 只展示目标、当前单元、待处理和进度；Study 只负责执行；Review 负责历史和调整。
4. 合并重复的“遇到问题”入口为统一 side chat / Learning Branch。
5. 区分“完成步骤、跳过步骤、暂缓任务、放弃任务、结束本次学习”。
6. 下一单元不可推进时按钮置灰；推进中显示明确动效和状态。
7. 修复所有页面的进度算法，统一读取正式状态投影。
8. 完成键盘焦点、颜色对比和窄窗口基础可访问性。

验收：5 名不了解内部实现的试用者能完成主流程；关键按钮不会造成概念误解；Today、Study、Review 对同一状态显示一致。

完成 P4 后达到 M2，综合成熟度目标约 85%。

### P5 长期使用体验

预计贡献：+5%。

任务：

1. U1 待处理中心：待评价、Guide 重试、冲突、计划确认。
2. U2 学习时间线：Task、Session、Submission、Evaluation、Review、Plan Change。
3. U3 知识库：按目标、状态、类型和出现次数筛选，并能查看来源证据。
4. 提供错题/误区复习入口；知识证据影响未来 proposal，不直接修改完成状态。
5. 学习风格设置只改变教学表达，不覆盖安全和完成标准。

验收：用户能回答“我学过什么、哪里卡住、系统为什么调整计划、下一步为什么是这个”。

### P6 兼容、备份与发布质量

预计贡献：+5%。

任务：

1. Q1 新 Session 完全使用 DailyGuideTask 锚点，旧 block/step 只读兼容。
2. 删除旧执行路径前完成数据映射和回滚方案，不删除用户数据。
3. U4 提供数据库导出、备份位置、恢复说明和版本兼容提示。
4. 建立 Electron 端到端流程：短目标、长目标、重启、跨日、失败重试、计划耗尽。
5. 真实 DeepSeek 合约测试保持 opt-in，但发布前人工执行完整主流程。
6. 增加 Token 成本统计：按 operation、模型、目标和日期汇总，不记录 secret。

验收：旧用户数据库升级后可继续学习；备份恢复可演练；端到端主流程无手工数据库操作。

### P7 连续真实使用验收

预计贡献：最后 5%，决定是否可称为高可用。

任务：

1. 使用真实 DeepSeek 连续运行至少 14 天、30 次 Focus Session。
2. 记录完成率、恢复成功率、AI schema 失败率、重复写入、错误推进和人工介入次数。
3. 每个问题只进入缺陷清单，不在试用中临时扩大产品范围。
4. P0/P1 缺陷必须清零；P2 缺陷有明确规避和修复计划。

高可用验收指标：

- 数据丢失：0。
- 错误自动推进：0。
- 重复 Evaluation/Guide：0。
- 需要手工修库：0。
- 应用重启恢复成功率：100%。
- 核心主流程完成率：≥95%。
- AI 失败后可见且可重试：100%。

完成 P7 后达到 M3，综合成熟度目标约 90%～95%。剩余 5% 属于持续产品优化，不是发布阻塞。

## 5. 每轮执行模板

每一轮只选择一个 Step，并按以下顺序推进：

1. 读取事实源和当前代码。
2. 写失败测试或恢复测试，确认真实断点。
3. 定义 Module Interface 和持久状态变化。
4. 迁移一条完整纵向调用链。
5. 删除被替代的重复实现。
6. 跑相关测试、全量测试、typecheck、build。
7. 更新本计划、`STEP_ROADMAP.md` 和必要的项目记忆。

禁止用“文件拆小了”“页面做出来了”“AI 能返回结果”作为完成依据。必须同时满足成功、失败、恢复、幂等和用户反馈。
