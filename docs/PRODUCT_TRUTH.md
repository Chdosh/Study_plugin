# Product Truth

状态：ACTIVE
用途：产品长期事实源。
优先级：用户当前显式请求优先于仓库内规则；当其他产品文档、历史设计或当前 UI 与本文件冲突时，以本文件为准。
限制：本文件只描述长期产品定位、核心概念、原则和废弃方案，不定义当前 MVP 范围、页面导航、数据字段、状态机跳转或验收测试。

## 1. 产品是什么

本产品是本地优先的 AI 学习教师，核心是帮助用户形成可恢复的学习闭环：澄清目标、拆解路径、陪伴执行、保存进度、评价结果，并在复盘后调整下一步。

AI 主动带学是核心；任务监督、计时器和浮窗只是辅助能力。用户应始终知道现在学什么、为什么学、做到什么算完成，以及卡住后如何继续。

## 2. 产品不是什么

本产品不是通用聊天工具、普通待办清单、番茄钟、日程表、强制监督工具或网页会话抓取器。它也不以复杂自治 Agent、社交排名、课程市场或强制锁屏作为产品方向。

## 3. 核心业务概念

* 学习目标：用户希望长期达成的学习结果，是计划和历史记录的顶层上下文。
* 阶段目标：学习目标下的阶段性方向，用于连接长期目标和当前重点。
* 主任务：需要完成并产生明确成果的学习任务，是正式提交和评价的中心单位。
* 行动步骤：主任务内部的过程引导，用于帮助用户推进，不等同于正式成果。
* Focus Session：一次实际执行记录，负责计时、暂停、恢复、提前结束和过程记录。
* 结果提交与评价：用户提交理解、练习、代码、截图或说明后，系统按任务类型给出评价和下一步建议。

一个主任务可以包含多个行动步骤，可以跨多个 Focus Session，也可以跨天完成。Focus Session 不等于任务成果，行动步骤不要求每一步都正式提交或调用 AI。

## 4. 产品级主流程

```text
创建或继续目标
→ 必要时澄清信息
→ 建立分层计划
→ 生成当前重点任务
→ 执行、提问和保存进度
→ 提交结果并获得评价
→ 复盘并调整后续计划
```

具体的默认展开范围、用户确认节点、页面展示方式和状态跳转由 `docs/MVP_SPEC.md` 描述。

## 5. 长期不能破坏的原则

* 分层目标与计划必须清晰，不能把长期目标、阶段重点和当前任务混成一段聊天文本。
* 当前任务应少量、完整、可执行，不为凑数量拆成大量碎片。
* 主任务可以跨多次执行恢复，不能把一次 Session 等同于任务完成。
* 用户提问后应能回到原任务和原执行位置。
* 结果提交形式应适配学习任务，可以是文字、代码、截图、运行结果、解释或自我验证。
* 评价应基于任务类型和用户提交，不要求所有任务采用统一成果格式。
* 目标、计划、任务、执行进度、提问上下文、提交、评价和复盘必须本地保存并可恢复。
* 历史记录应保留目标变化、计划版本、任务进度、执行记录、提交结果和复盘。
* 复盘按任务、阶段或用户需要触发，不应无条件固定发生。
* 任务切换、暂缓、归档和恢复是核心能力，不能为了简化 UI 删除。

## 6. 已废弃的历史方案

以下方案不再作为当前产品的默认规则或核心流程：

* 固定十分钟任务。
* 按固定时长把一天拆成大量碎片任务。
* 每完成一个普通微步骤都调用一次 AI。
* 强制用户确认所有普通任务变化或细小计划变化。
* 固定每天无条件复盘。
* 将一次 Focus Session 等同于一个任务。
* 用浮窗或专注模式替代完整主页面。
* 为简化 UI 删除历史、提交、恢复、暂缓、切换或计划入口。

## 7. 产品级 Non-Goals

* 抓取 ChatGPT、Codex 或其他网页会话。
* 监控手机使用。
* 截图、录屏、键盘记录、收集私信内容或强制锁屏。
* 社交排名、课程市场或以竞争机制驱动学习。
* 未经确认让 AI 永久修改用户计划、数据或画像。

## 8. 学习日推进机制

分层计划由长期 roadmap、短期 shortPlan（通常为前几天的粗略安排）和每日 dailyGuide 组成。学习日按以下规则推进：

1. **daily guide 必须通过 AI 生成**。不得从 shortPlan 的字符串任务本地拼接为 DailyGuideTask。每次生成 daily guide 都调用 `dailyGuideAgent`，由 AI 根据当前阶段和前一日结果生成结构化主任务与执行动作。

2. **完成当天任务后，当前 daily guide 进入 completed 状态**，并保留供 Today 和 Review 页面读取，不因归档而丢失。已完成的学习日数据是持久历史，不得覆盖或删除。

3. **完成后不得因查不到 active guide 而退回目标访谈入口**。Today 页面在无 active guide 且存在已完成 guide 时，应展示已完成日的总结，而不是触发重新访谈。

4. **下一学习日开始**时，从当前 goal 的 shortPlan 中选择 `date === null` 且 `dayIndex` 最小的 `ShortPlanDay` 作为目标学习日。

5. **激活该学习日**时，将选中 `ShortPlanDay.date` 设为当前实际日期，标记该日已进入执行阶段。

6. **调用 `dailyGuideAgent`** 生成该学习日的执行稿。AI 输入应包含：当前 roadmap 阶段、目标 `ShortPlanDay`、前一天完成情况（若存在）、submission evaluation 结果、review 结果。

7. **AI 失败处理**：若 `dailyGuideAgent` 调用失败（schema 校验失败、超时、网络错误等），保留目标 `ShortPlanDay` 的 `date` 为当前日期，但其对应的 daily guide 标记为 `draft` 且处于待生成状态。允许用户手动重试，不得修改或删除已完成学习日的数据，不得跳过该学习日直接推进到下一个 dayIndex。

8. **当前版本限制**：默认每天只激活一个 `ShortPlanDay`。不实现在同一天内连续推进多个 dayIndex 的机制。若用户在同一天内完成所有任务后再次触发"下一天"，应生成当天的第二个 daily guide（覆盖原当日 guide），或引导用户等待次日。
