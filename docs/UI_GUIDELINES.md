# UI Guidelines

本文是当前业务 UI 和交互重构的活跃专题文档。`docs/UI_SYSTEM.md` 可作为历史设计系统细则参考，但正式业务页面的优先规则以本文、当前代码和 `docs/PROJECT_MEMORY.md` 为准。

## 1. 基础规则

UI 重构首先解决信息层级、页面职责和用户下一步行动，不优先通过配色、圆角、阴影或装饰提升来掩盖结构问题。

通用规则：

* 每个页面或弹窗最多突出一个主操作。
* 主操作必须对应用户当前最重要的下一步。
* 复用现有组件和 design token，避免创建近似重复组件。
* 使用稳定一致的间距、字号、圆角、边框和交互状态。
* 使用正式图标，不使用 emoji 作为 UI 图标。
* 避免过度渐变、玻璃拟态、发光、复杂阴影和装饰动画。
* 不要把每个内容组都放进卡片。
* 优先用排版、间距、分隔线和信息顺序表达层级。
* 颜色主要用于层级、状态和当前选中，不用于给每个模块制造存在感。
* 保持可读对比度。
* 优先渐进披露，不要一次展示所有选项。
* 破坏性操作必须视觉上区分，并在不可逆时要求确认。
* 重要操作应尽量支持键盘触达和清晰 focus 状态。

用户可见应用 UI 使用中文。避免中英文混杂的 UI 标签，除非英文术语必要或已被广泛使用。

## 2. 页面状态

每个重要页面或组件都应考虑：

* 加载状态。
* 空状态。
* 错误状态。
* 禁用状态。
* 部分数据状态。
* 离线或 AI 不可用状态。

用户应始终能识别：

* 当前任务是什么。
* 当前系统状态是什么。
* 当前可执行动作是什么。
* 执行动作后会发生什么。
* 变更是否已经保存。
* AI 结果是否只是建议。

## 3. UI 重构边界

UI-only 任务必须遵守：

* 保留当前业务行为。
* 不修改数据库 schema。
* 不重写无关 service、IPC 或 AI 流程。
* 不新增 speculative feature。
* 不移除既有可用行为来简化视觉实现。
* 视觉重构后验证完整用户流程。

如果 UI 调整会影响数据持久化、IPC、AI 决策、用户确认规则或现有学习闭环，必须把它视为产品/架构变更，而不是单纯 UI 任务。

## 4. 当前任务的视觉中心

当前页面的核心问题是：最重要的任务没有形成绝对视觉中心，同一信息在多个区域重复展示，卡片嵌套过多，关键操作经常位于首屏以下。

页面应按照用户下一步行动组织，而不是按照数据库字段逐项展示。数据库里有目标、产出、标准、fallback 和其他任务，不代表这些内容都必须同时以独立卡片出现。

推荐全局框架：

```text
固定左侧导航
    |
    |-- 页面主任务区：用户现在需要完成的事情
    |-- 右侧辅助区：验收、AI、提交、补充信息
```

不要继续通过添加更多卡片来解决页面混乱。成熟页面应把主任务区和辅助信息区分清楚，让用户进入页面后不需要重新判断“现在该看哪里”。

## 5. 全局布局与响应式

桌面宽屏建议采用：

* 左导航：216px。
* 主内容：`minmax(640px, 820px)`。
* 辅助栏：320px。
* 列间距：24px。
* 页面左右边距：32px。
* 内容总宽度：不超过 1400px。

响应式规则：

* 窗口宽度大于等于 1440px：216px 导航 + 主内容 + 320px 辅助栏。
* 窗口宽度 1100px 到 1439px：72px 窄导航 + 主内容 + 300px 辅助栏。
* 窗口宽度小于 1100px：辅助栏变成右侧抽屉，不持续占用页面宽度。
* 窗口宽度小于 800px：左侧导航变成覆盖式菜单，页面只保留一个主内容栏。

布局不应只是同比缩小，而应在断点处改变窗格数量、位置和可见内容。

## 6. 今日页

今日页的职责是：看清今天做什么，并启动当前任务。它不应把计划确认页、任务概览页和任务执行入口合并成一个无限长页面。

首屏必须让用户看到：

* 今日目标。
* 当前任务。
* 唯一主按钮。

建议信息结构：

```text
今日
路径：目标 / 当前阶段 / 今日

今日目标                                      1/6 · 120分钟
完成 Git 基础操作与两次本地提交
[开始当前任务]

当前任务                                      20分钟 · 待开始
环境准备：安装与配置 Git

目标
确认 Git 可用并配置身份信息

操作
1. 检查 Git 版本
2. 配置 user.name 和 user.email
3. 检查配置结果

产出 · 完成标准
...

[开始任务]

今日其他任务
20:20  创建练习仓库
20:40  完成首次提交
21:00  完成第二次提交
```

右侧辅助栏只保留：

* 今日进度。
* 结束验收。
* 学习边界。

今日页规则：

* 合并“确认今日执行稿”和“开始当前任务”。未确认状态下首屏只显示“确认并开始”；确认后变为“开始当前任务”。
* 同一页面或同一信息区不应同时出现多个争夺注意力的主按钮。
* 删除当前任务内部的嵌套卡片。当前任务本身已经是一个内容对象，内部用标题、间距和分隔线组织。
* 当前任务内部顺序为：标题和状态、目标、操作、产出与完成标准、卡住时怎么办。
* “卡住时怎么办”默认折叠。
* “今日其他任务”不要再次显示当前任务，列表只从下一项开始。
* “明天前五个动作”不常驻今日页；它可以在复盘结束后作为“明日预告”，或折叠在右栏底部。

## 7. 学习页

学习页的职责是：专注完成当前步骤，提问和提交。它必须形成真正的任务工作区，避免任务标题、计时器、步骤条、AI 面板和悬浮窗同时成为视觉中心。

建议信息结构：

```text
创建 README.md 并完成首次提交        步骤 2/5        00:22
                                              [暂停] [结束]

主要任务区                                      AI 助手

当前步骤                                        [提问] [提交结果]
创建 README.md 并首次提交

具体操作                                        当前步骤摘要
1. ...                                          确保 Git 可用并配置身份...
2. ...

预期产出                                        输入区域
完成标准

后续步骤
```

学习页规则：

* 计时器属于当前任务状态，应放入顶部固定 Session Bar，而不是单独成为内容块。
* Session Bar 同时承载当前任务短标题、步骤进度、当前计时、暂停和结束。
* 当前任务标题使用 24px 到 28px，避免长中文标题在窄区域连续换行。
* 取消横向长文本步骤条，改为纵向或折叠列表。
* 当前步骤默认展开，其他步骤只显示标题。
* 右侧 AI 侧栏不重复展示主页面已有的当前步骤内容。
* AI 侧栏可采用“问题与回答 / 提交与评估”两个标签，或直接展示当前上下文、提问入口、提交结果入口和输入框。

推荐字号层级：

| 用途 | 字号 / 行高 |
| --- | --- |
| 页面标题“学习” | 28 / 36 |
| 当前任务标题 | 24 / 32 |
| 模块标题 | 20 / 28 |
| 小标题 | 16 / 22 |
| 正文 | 14 / 20 |
| 辅助信息 | 12 / 16 |
| 命令代码 | 13 / 20 等宽字体 |

## 8. 悬浮窗

悬浮窗默认必须是紧凑状态，不能用大窗口展示极少内容。

收起状态：

```text
● 配置 Git 身份信息        00:32        [暂停] [展开]
```

收起尺寸建议：

* 宽度：约 420px。
* 高度：52px 到 60px。

展开状态才显示：

* 当前步骤。
* 具体操作摘要。
* 完成标准。
* 快速提问。
* 标记完成。

展开尺寸建议：

* 宽度：约 420px。
* 高度：280px 到 360px。

展开状态不能是空白窗口，并且应保存用户最后一次展开状态。普通页面卡片使用边框即可，阴影主要保留给悬浮窗、抽屉和弹层。

## 9. 主动访谈页

主动访谈页的职责是：通过单一聊天工作区澄清目标，并在信息足够时让用户确认目标理解。

不要把首次启动说明、AI 聊天、用户输入、目标理解表单和生成计划按钮都放进同一条纵向滚动流。

建议信息结构：

```text
主动访谈

AI：你准备学习什么？

用户：一天学会 Git 基础命令

AI：你目前有什么基础...

输入内容...                              [直接开始]
                                         [发送]
```

主动访谈页规则：

* 输入区固定在底部，不随着聊天记录滚走。
* 当 AI 认为信息足够时，右侧出现“目标理解”辅助栏。
* 不要在聊天记录下方突然插入大型表单。
* 用户点击“修改”后，再把目标理解摘要字段变成输入框。
* 右侧“本轮会生成什么”不长期存在；可以缩成顶部一句状态，或仅在首次访谈时显示一次。

目标理解辅助栏建议结构：

```text
目标理解

目标
一天掌握 Git 基础命令

产出
建立仓库并完成两次提交

时间
今天，约 5 小时

[修改]
[确认并生成计划]
```

## 10. Design Token 与视觉约束

先建立少量稳定 token，再逐页应用，不要继续逐个页面手调。

建议基础 token：

```css
--space-1: 4px;
--space-2: 8px;
--space-3: 12px;
--space-4: 16px;
--space-5: 24px;
--space-6: 32px;

--radius-control: 6px;
--radius-card: 10px;
--radius-overlay: 12px;

--page-padding: 32px;
--card-padding: 20px;
--section-gap: 24px;
--control-height: 36px;
--primary-button-height: 40px;
```

颜色只保留四种语义：

* 品牌强调色：主按钮、当前状态、选中导航。
* 中性背景色：页面背景。
* 中性表面色：卡片和面板。
* 状态色：成功、警告、错误。

品牌色只用于：

* 当前导航。
* 主按钮。
* 当前任务标识。
* 关键进度。

灰色正文和浅色边框需要进行对比度检测。交互目标至少应满足 WCAG 2.2 的 24px × 24px CSS 像素最低尺寸，并提供清晰的键盘焦点状态。

Git 命令不要混在普通加粗正文中，统一使用等宽代码块，并在右侧提供复制按钮：

```bash
git config --global user.name "你的名字"
git config --global user.email "你的邮箱"
```

## 11. 实施优先级

P0：先改变结构，不做精装修。

* 重构今日页首屏，让“今日目标、当前任务、开始按钮”全部在首屏可见。
* 删除当前任务内部的嵌套卡片，改用标题、间距和分隔线。
* 当前任务不再出现在“今日其他任务”列表。
* 重构学习页：顶部固定 Session Bar，主区域展示当前步骤，右栏只放 AI 和提交。
* 横向步骤条改成纵向或折叠列表。
* 悬浮窗默认改成约 420px × 56px 的紧凑条。

P1：统一设计系统。

* 建立字号、间距、圆角、颜色和边框 token。
* 所有页面使用同一列宽和顶部对齐线。
* 主按钮每个页面只保留一个。
* 为 Git 命令增加代码块和复制功能。
* 支持右侧辅助栏在窄窗口变成 Drawer。

P2：最后再做视觉提升。

* 悬停效果。
* 页面切换动画。
* 微弱阴影。
* 空状态插图。
* 图标统一。
* Mica 或半透明效果。

在结构未稳定前，不应优先更换渐变、增加阴影或重画图标。结构不变的情况下，这些只会得到更精致但依然混乱的页面。

## 12. 完成检查

页面完成前必须确认：

* 用户能在 3 秒内识别当前主操作。
* 首屏只有一个占主导地位的动作。
* 生产页面不显示开发控制器或状态切换工具。
* 主视口内不出现超过三个带边框容器，除非内容确实需要。
* 相关文本优先通过间距和排版组织，而不是继续加卡片。
* 状态色只在对应状态真实发生时出现。
* 页面不使用多个品牌强调色。
* 辅助信息的视觉权重明显低于当前任务。
* 当前任务、当前状态和下一步动作可以立即识别。
* 页面已通过真实应用截图或实际运行界面检查，而不只看源码。

最终产品分工：

* 今日：看清今天做什么，并启动任务。
* 学习：专注完成当前步骤，提问和提交。
* 复盘：查看结果，决定是否调整。
* 设置：模型与应用配置。

每个页面只有一个核心问题，界面才会真正稳定。
