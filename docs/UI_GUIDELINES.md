# UI Guidelines

状态：ACTIVE
用途：当前 UI 与交互设计规则。

## 1. 文档边界

本文只定义信息架构、视觉层级、交互呈现和 UI 检查方式。产品行为、功能边界、AI 调用、任务完成、计划调整和持久化规则以 `docs/PRODUCT_TRUTH.md` 与 `docs/MVP_SPEC.md` 为准。

事实源顺序：

1. 用户当前明确要求。
2. `docs/PRODUCT_TRUTH.md`。
3. `docs/MVP_SPEC.md`。
4. 本文。
5. 已确认的技术设计。
6. 当前代码。
7. `docs/PROJECT_MEMORY.md` 和历史方案。

当前代码用于确认实现现状，不作为产品或 UI 规则的最高事实源。`docs/PROJECT_MEMORY.md` 仅用于了解历史背景，冲突时不覆盖活跃规范。

本文中的“开始”“提交”“评价”“复盘”等文案只描述界面呈现，不定义业务触发条件。是否需要用户确认、何时调用 AI、任务如何完成、计划如何调整，以 `docs/MVP_SPEC.md` 为准。

## 2. 通用 UI 原则

UI 重构首先解决信息层级、页面职责和用户下一步行动，不优先通过配色、圆角、阴影或装饰提升来掩盖结构问题。

通用规则：

* 每个页面状态最多只有一个视觉主操作。其他必要操作使用次级按钮、文字按钮、菜单或危险操作样式，不与主操作争夺视觉权重。
* 页面状态变化后，视觉主操作可以变化。访谈进行中主操作是“发送”，等待目标理解确认时主操作才是“确认并生成计划”。Today 不承载开始、暂停、继续、提交等学习操作；Study 页的主操作随状态变化为“开始学习”“完成当前步骤”或“提交结果”，暂停和提问是次级操作。
* 主操作必须对应用户当前最重要的下一步。
* 优先复用职责清晰、样式稳定的现有组件和 design token。对职责混乱、重复样式严重或仅服务旧页面结构的组件，应先盘点依赖，再小范围替换。
* 使用稳定一致的间距、字号、圆角、边框和交互状态。
* 使用正式图标，不使用 emoji 作为 UI 图标。
* 避免过度渐变、玻璃拟态、发光、复杂阴影和装饰动画。
* 不要把每个内容组都放进卡片，优先用排版、间距、分隔线和信息顺序表达层级。
* 颜色主要用于层级、状态和当前选中，不用于给每个模块制造存在感。
* 优先渐进披露，不要一次展示所有选项。
* 破坏性操作必须视觉上区分，并在不可逆时要求确认。

用户可见应用 UI 使用中文。避免中英文混杂的 UI 标签，除非英文术语必要或已被广泛使用。

## 3. 页面状态

每个重要页面或组件都应考虑：

* 加载状态。
* 空状态。
* 错误状态。
* 禁用状态。
* 部分数据状态。
* 离线或 AI 不可用状态。
* AI 生成中状态。

用户应始终能识别：

* 当前任务是什么。
* 当前系统状态是什么。
* 当前可执行动作是什么。
* 执行动作后会发生什么。
* 变更是否已经保存。
* AI 结果是否只是建议。

AI 生成期间应保留用户已输入内容，明确显示正在生成的对象，并在失败后提供重试入口。AI 建议与已正式保存的数据应视觉区分。重试、保存、离开页面等数据一致性规则由 `docs/MVP_SPEC.md`、架构文档或具体实现约束定义。

## 4. UI 重构边界

UI-only 任务必须遵守：

* 保留当前业务行为。
* 不修改数据库 schema。
* 不重写无关 service、IPC 或 AI 流程。
* 不新增 speculative feature。
* 不移除既有可用行为来简化视觉实现。
* 视觉重构后验证相关用户流程。

如果 UI 调整会影响数据持久化、IPC、AI 决策、用户确认规则或现有学习闭环，必须把它视为产品/架构变更，而不是单纯 UI 任务。

## 5. 页面与布局规则

页面应按照用户下一步行动组织，而不是按照数据库字段逐项展示。数据库里有目标、产出、标准、fallback 和其他任务，不代表这些内容都必须同时以独立卡片出现。

Today 和 Study 等任务型页面默认采用左侧导航、主任务区和可选辅助区：

```text
左侧导航
    |
    |-- 页面主任务区：用户现在需要完成的事情
    |-- 可选辅助区：验收、AI、提交、补充信息
```

其他页面根据职责使用单栏或双栏结构，不强制保留空置右栏。设置页、复盘详情、历史记录、空状态和首次引导可以采用不同布局。

布局参数是当前桌面实现建议，可根据真实窗口截图、最小窗口尺寸、Windows 缩放比例、标题长度和实际内容调整，不属于永久业务规则：

* 左导航：约 216px。
* 主内容：`minmax(640px, 820px)`。
* 辅助栏：约 300px 到 320px。
* 列间距：24px。
* 页面左右边距：32px。
* 内容总宽度：不超过 1400px。

当侧边导航已经明确当前页面时，内容区不重复显示“今日”“学习”等大型页面名称。内容区最高层标题优先使用当前目标或当前主任务标题。

### Today

Today 的职责是看清今天做什么、当前进度如何、知识库有什么沉淀，以及管理今日计划。Today 不启动当前任务；开始、暂停、继续、完成步骤和提交结果都属于 Study。

建议信息结构：

```text
路径：目标 / 当前阶段 / 今日

今日目标                                  任务 1/4 · 今日预计 120 分钟
完成 Git 基础操作与两次本地提交

当前任务                                  约 30 分钟 · 待开始
环境准备：安装与配置 Git

目标
确认 Git 可用并配置身份信息

操作
1. 检查 Git 版本
2. 配置 user.name 和 user.email
3. 检查配置结果

产出与完成标准
...

[确认今日执行稿]

今日其他任务
创建练习仓库                             约 30 分钟 · 待开始
完成首次提交                             约 40 分钟 · 待开始
完成第二次提交                           约 45 分钟 · 待开始
```

只有用户明确设置了具体日程时，才显示绝对时间。默认不要用固定时刻组织今日任务，避免重新引入固定 Time Block。

“今日其他任务”不要再次显示当前任务。当前任务内部不要嵌套多层卡片，使用标题、间距和分隔线组织目标、操作、产出与完成标准。

### Study

Study 的职责是开始当前任务、专注完成当前行动步骤、提问和提交。计时器属于当前任务状态，应放入顶部 Session Bar，而不是单独成为内容块；当前 MVP 不提供独立“结束计时/结束学习”入口，主任务提交评价通过后同步结束当前计时。

建议信息结构：

```text
创建 README.md 并完成首次提交        当前行动 · 已完成 1 项        00:22
                                              [暂停]

主要任务区                                      AI 助手

当前行动步骤                                    [提问] [提交结果]
创建 README.md 并首次提交

具体操作                                        当前上下文
1. ...
2. ...

预期产出
完成标准

后续行动
```

当任务确实拥有稳定行动数量时，可以显示“行动 2/5”。如果行动步骤会动态生成、调整或跳过，使用“当前行动”“已完成 N 项”等更稳妥表达。

右侧 AI 侧栏只显示一行当前上下文摘要，不重复完整步骤说明、操作列表和完成标准。

### 主动访谈

主动访谈页的职责是通过单一聊天工作区澄清目标，并在信息足够时让用户确认目标理解。输入区固定在底部，不随着聊天记录滚走。

建议信息结构：

```text
AI：你准备学习什么？

用户：一天学会 Git 基础命令

AI：你目前有什么基础...

输入内容...                              [发送]
                                         使用当前信息生成初步计划
```

“发送”是访谈进行中的视觉主按钮；“使用当前信息生成初步计划”是次级文字操作，具体是否允许跳过更多访谈以 `docs/MVP_SPEC.md` 为准。等待目标理解确认时，主按钮才变为“确认并生成计划”。

### 独立浮窗与页面跳转

当前 MVP 不提供独立学习浮窗。计时、暂停、恢复、完成步骤、提问和提交都属于 Study 页；不要在托盘、浮窗、复盘页、设置页或总览页重复提供这些学习控制。

页面内部不要用“返回总览”“开始下一次学习”“查看全部历史”等按钮制造互相跳转。页面切换由全局导航负责；各页面只保留本页职责内的操作。

## 6. Design Token 和可访问性

先建立少量稳定 token，再逐页应用，不要继续逐个页面手调。

建议基础 token：

```css
--space-1: 4px;
--space-2: 8px;
--space-3: 12px;
--space-4: 16px;
--space-5: 24px;
--space-6: 32px;

--radius-control: 6px;
--radius-card: 10px;
--radius-overlay: 12px;

--page-padding: 32px;
--card-padding: 20px;
--section-gap: 24px;
--control-height: 36px;
--primary-button-height: 40px;
```

颜色使用语义 token，不在页面组件中大量硬编码颜色值。至少区分品牌强调、背景、表面、文本、边框、状态、hover、focus、disabled 和遮罩层。

品牌色只用于当前导航、主按钮、当前任务标识和关键进度。灰色正文和浅色边框需要进行对比度检测。

命令和代码使用等宽代码块，适合复制的内容提供复制按钮。

基础可访问性要求：

* 不能只依赖颜色表达状态。
* 图标按钮必须有文字提示或可访问名称。
* 键盘 Tab 顺序应与视觉顺序一致。
* 重要操作应支持键盘触达和清晰 focus 状态。
* 交互目标至少应满足 WCAG 2.2 的 24px × 24px CSS 像素最低尺寸。
* 动画应支持减少动态效果。
* 文本放大后不能遮挡核心操作。
* 禁用按钮需要说明原因，不能只变灰。

## 7. 完成检查

页面完成前建议确认：

* 通过真实截图或实际运行检查，首次进入页面时应能快速识别当前任务、当前状态和主操作。
* 生产页面不显示开发控制器或状态切换工具。
* 主视口中的边框容器职责明确，没有大量同权重卡片互相竞争。
* 状态色只在对应状态真实发生时出现，页面不使用多个品牌强调色。
* 辅助信息的视觉权重明显低于当前任务。
* 页面已覆盖关键加载、空、错、禁用、离线或 AI 不可用状态。
* 页面已通过真实应用截图或实际运行界面检查，而不只看源码。

当前 MVP 页面分工：

* 今日：看清今天做什么、进度和知识库，并管理计划。
* 学习：开始任务、专注完成当前行动步骤，提问和提交。
* 复盘：查看结果，决定是否调整；不作为学习流程中强制弹出的独立结算页。
* 设置：模型、隐私与应用配置。

每个页面只有一个核心问题，界面才会真正稳定。
