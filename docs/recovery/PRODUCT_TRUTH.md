# 产品真相草案

> 本文件只用于恢复产品方向，不代表最终规范。现有文档可能互相冲突，需经用户确认后再写回正式文档。

## 1. 核心问题

用户不是想要一个通用聊天工具或普通待办清单，而是需要一个本地优先、可长期陪伴的 AI 学习教师：能帮助澄清目标、拆成可执行学习路径、陪用户进入当天学习、记录过程、评估结果，并在复盘后调整下一步。

核心难点不是“生成计划”，而是让用户在多天、多轮学习中始终知道：现在学什么、为什么学、做到什么算完成、卡住时怎么继续、今天结束后如何保留进度。

产品核心定位是“AI 老师主动带学”，任务监督只是辅助能力。它不是普通待办清单、番茄钟、日程表或普通聊天软件。

## 2. 核心概念

* 主任务：当天需要完成并产生明确成果的学习任务，是最小闭环的中心。
* 行动步骤：主任务内部的过程引导，用于告诉用户怎么推进。
* Focus Session：一次实际执行记录，负责计时、暂停、恢复、提前结束和记录过程。

一个主任务可以包含多个行动步骤，也可以经历一次或多次 Focus Session。任务时长根据内容动态确定，不固定为十分钟。正式成果提交和 AI 评价以主任务为单位；行动步骤可以轻量确认，但不要求每一步正式提交或调用 AI；Focus Session 不等于任务成果。

## 3. 完整主流程

1. 用户创建、选择或继续一个学习目标。

2. 当目标信息不足时，AI 根据已有信息进行简短主动追问，澄清目标、当前基础、时间条件和预期结果。已有信息足够时，不重复完整访谈。

3. AI 建立或更新分层计划，包括长期方向、阶段目标和近期重点。首次建立计划或发生重大方向调整时，由用户确认。

4. 系统根据当前进度生成少量、范围完整的今日重点任务。普通今日任务不强制逐次确认，用户可以直接开始、修改或要求重新生成。

5. Today 首屏以当前最值得执行的任务为视觉中心，同时提供今日任务概览、阶段位置和长期目标入口。具体采用侧栏、折叠或其他布局，由 UI 设计决定。

6. 用户进入学习执行过程。一个主任务可以包含若干行动步骤，也可以跨越多次执行记录完成。

7. AI 根据当前任务逐步给出必要的行动指引。用户可以完成当前动作、记录结果、提出问题，AI 回答后继续回到原任务主线。

8. 用户提交当前阶段的理解、练习结果、代码、截图、文字说明或其他产出。验证形式根据任务类型决定，不要求所有任务都有统一的最终成果格式。

9. AI 根据当前结果判断：继续当前步骤、补充解释、修改结果、进入下一步骤、完成当前任务，或在必要时调整后续计划。

10. 用户可以随时暂停或结束本次执行，并在之后从已保存的状态继续。本次执行结束不等于主任务完成。

11. 当主任务、学习阶段或当天学习确实结束时，系统可以生成复盘，总结进度、问题和下一步。复盘按任务需要或用户主动触发，不强制每天固定发生。

12. 用户可以暂缓、归档或切换任务。只有创建新目标、目标发生明显变化或现有信息不足时，AI 才重新进行必要的主动访谈。

## 4. 应保留或恢复的核心能力

* 按需进行主动目标访谈，并复用已有用户信息，避免重复提问。
* 长期目标、阶段目标、近期重点和当前任务之间具有清晰层级。
* 每天或每个学习周期生成少量但内容完整的重点任务。
* 当前重点任务突出展示，同时能够查看今日概览和长期计划位置。
* 学习任务内部可以逐步推进，并根据用户结果动态给出下一步。
* 学习过程中可以随时提问，回答后能够回到原任务和原步骤。
* 开始、暂停、继续、暂时结束和恢复执行。
* 一个主任务可以跨多次执行完成。
* 支持文字、代码、截图、运行结果、解释或自我验证等多种结果提交方式。
* AI 根据任务类型评估当前结果，并决定继续、修改、补充或进入下一步。
* 本地保存目标、计划、任务、执行进度、提问上下文和评估结果，应用重启后能够恢复。
* 支持任务暂缓、归档、切换和恢复。
* 保留历史记录，包括目标变化、计划版本、任务进度、执行记录、提交结果和复盘。
* 在主任务、阶段或用户主动结束学习时进行复盘，并在必要时调整后续计划。
* 浮窗或专注模式作为执行辅助，不能替代完整主页面。
* 产品坚持本地优先和隐私优先原则。

界面需要覆盖今日重点、学习执行、计划与历史、复盘和设置等能力，但当前阶段不锁死导航名称、页面数量和具体布局。

## 5. 不再作为当前强制规则的历史方案
以下方案不再作为当前产品的默认规则或核心流程。相关概念可以在适当场景保留，但不得继续采用固定化、强制化或替代核心流程的实现方式。现有代码在完成依赖盘点前不得直接删除。

* 将所有任务固定为十分钟。
* 按固定时长把一天拆成大量碎片任务。
* 每完成一个普通微步骤都必须调用一次 AI。
* 强制用户逐次确认所有今日任务和细小计划变化。
* 将每日复盘设为无条件发生的固定流程。
* 将一次执行记录等同于一个任务。
* 要求所有学习任务采用统一的最终成果形式。
* Today 首屏展示大量同等级任务或复杂完整日程。
* 用浮窗替代主页面完整流程。
* 为简化 UI 删除历史、结束、恢复、提交或计划入口。

## 6. 文档冲突或需要重新审查的规则

* Task、Step、Action、Checkpoint、Block、Session 等术语曾被多次重命名，不能只进行文字替换，应先确认各自是否具有独立业务含义。
* “主任务、行动步骤、执行记录”是核心区别，但学习目标、阶段目标、今日重点和结果验证也需要保留独立概念。
* 早期文档中的时间块和学习块可能混合了任务内容、预计时长和执行记录，需要分别审查。
* 最近一次 UI 重构只代表一种展示方案，不能作为产品流程和功能取舍的最终事实。
* 历史文档中的部分实现可能仍承载必要功能，不能仅因名称旧而删除。
* 复盘应根据任务、阶段和用户行为触发，不预设固定的每日调用次数。

## 7. 仍需后续确认的问题

* AI 在任务内部应主动推进到什么程度，什么情况下必须等待用户操作？
* 普通行动步骤、关键验证节点和主任务完成，分别需要多强的结果验证？
* 当用户同时存在多个学习目标时，系统如何选择或推荐当前重点？
* 历史文档和旧版本中，是否还有未列入本文件但必须恢复的功能？

## 8. 当前恢复阶段不应继续做

* 不在产品事实尚未确认时继续大规模重构业务代码。
* 不继续增加职责重叠的规范文档。
* 不优先进行配色、阴影、动画和局部布局精修。
* 在完成现状盘点和迁移方案前，不随意新增、删除或重构数据库结构、AI 流程和核心类型。
* 不直接删除旧表、旧文档、旧组件或旧功能入口。
* 不把最近一次 UI 重构后的页面结构直接认定为最终产品结构。
* 云同步、复杂 RAG、知识图谱、扩展监控和多 Agent 开发框架不属于当前恢复阶段。
