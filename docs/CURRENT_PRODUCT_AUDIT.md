# 当前产品审计

日期：2026-06-20
审计范围：`AGENTS.md`、`docs/PROJECT_MEMORY.md`、当前 `src/main`、`src/preload`、`src/renderer`、`src/shared` 代码。
审计方式：静态代码审查，未运行应用、未执行测试、未重新设计页面。

## 0. 总体判断

当前软件已经具备一个本地优先学习管家的最小技术闭环：配置 DeepSeek、导入学习计划、解析任务、生成日计划草稿、确认计划、开始学习块、记录前台应用变化、完成或跳过学习块、生成复盘。

但从产品流程看，当前仍像“功能原型”而不是稳定的学习监督系统。核心问题不是页面数量太多，而是状态规则、下一步引导、确认边界、历史版本和执行状态之间没有形成统一的产品语义。用户可以操作很多按钮，但系统没有始终明确回答“我现在应该做什么、为什么做、做完后发生了什么”。

## 1. 当前所有页面、弹窗和入口

### 1.1 Electron 入口

- 主窗口：`BrowserWindow`，标题为“学习管家”。
- 窗口关闭行为：点击关闭不会退出应用，而是隐藏窗口并保留托盘应用。
- 托盘入口：
  - “打开学习管家”：显示主窗口。
  - “开始当前学习块”：显示主窗口并弹系统通知，但不会真正开始任何学习块。

  - 托盘“开始当前学习块”提示用户打开“今日计划”。

## 2. 每个页面目前承担的用户目标

### 学习工作台

当前目标：

- 把外部学习计划导入为本地任务。
- 使用 AI 生成今日草稿计划。
- 查看今日生成过的计划历史。
- 确认草稿计划。
- 开始、跳过、完成学习块。

问题：

- 它同时承担导入、计划、执行、历史、专注记录五类目标，当前缺少明确步骤提示。
- “今日上下文”的概念出现在按钮文案里，但没有可视化展示上下文里具体有哪些任务、来源、目标。
- “生成今日草稿”使用所有未完成任务，而不是用户明确选中的今日上下文。

### 任务清单

当前目标：

- 查看本地任务。
- 手动修改任务状态。

问题：

- 不能创建任务。
- 不能编辑任务标题、描述、估时、验收标准。
- 不能查看目标、依赖关系、来源导入文本。
- 与学习工作台之间没有明确的“任务被纳入今日计划”状态联动。

### 复盘

当前目标：

- 生成 AI 日复盘。
- 查看完成度、专注度、总结和下一步动作。

问题：

- “人工确认”只有说明，没有实际确认或拒绝流程。
- 复盘结果保存在 `ai_reviews`，但 UI 不读取历史复盘。
- 复盘不会产生可审查的重排建议。
- 复盘页面不能解释分数来源。

### 设置

当前目标：

- 配置 DeepSeek。
- 配置默认学习块时长。
- 编辑 prompt profile。

问题：

- `autoLaunch` 后端存在，但 UI 没有开关。
- `dailyStudyWindows` 后端存在，但 UI 没有编辑入口。
- Prompt profile 编辑入口和普通设置混在一起，风险较高。
- 保存设置和保存提示词是两个不同动作，但视觉区域相邻，容易误解。

## 3. 当前完整用户操作流程

### 3.1 新用户理想流程

1. 打开应用。
2. 进入学习工作台。
3. 看到 DeepSeek 未配置。
4. 进入设置。
5. 填写 API Key、base URL、model，点击保存。
6. 返回学习工作台。
7. 粘贴 ChatGPT/Codex/手动计划。
8. 选择来源和提示词。
9. 点击“导入到今日上下文”。
10. AI 解析导入内容。
11. 本地创建 goals、tasks、dependencies。
12. 工作台刷新，未完成任务数增加。
13. 点击“生成今日草稿”。
14. AI 根据未完成任务生成计划块。
15. 本地保存 draft daily plan、daily plan blocks、plan version、AI review。
16. 中栏展示新草稿。
17. 右栏历史出现新版本。
18. 用户点击“确认草稿”。
19. 计划状态变成 confirmed。
20. 用户点击某个学习块的“开始”图标。
21. 本地创建 study session，计划块状态变成 active。
22. Focus monitor 每 15 秒记录前台应用变化。
23. 用户填写备注，点击“完成”。
24. 本地结束 session，学习块变成 done，关联 task 直接变成 done。
25. 用户进入复盘页，点击“生成”。
26. AI 根据 day snapshot 生成复盘。
27. UI 展示复盘结果。

### 3.2 当前实际流程中的断点

- 用户首次打开时，缺 API Key 只显示顶部状态，不会直接引导到设置。
- 导入失败后，raw import 仍可能停留在 created 状态，UI 没有导入历史或失败恢复入口。
- 计划生成使用所有未完成任务，用户无法选择“今天只规划哪些任务”。
- 确认计划不会归档旧的 confirmed 或 draft 计划，今天可能同时存在多个 confirmed 计划。
- 开始按钮对所有块可见，不区分 planned、active、done、skipped。
- 完成 session 后 renderer 仍保留非空 `activeSession`，当前专注块可能继续显示已完成 session。
- 完成一个计划块会把关联 task 直接标记为 done，不支持同一任务拆成多个学习块。
- 复盘使用所有 sessions，而不是按日期过滤 session。
- 复盘结果没有确认、应用、拒绝或重排路径。

## 4. 重复功能、冲突功能和入口过多的问题

### 4.1 重复或语义重叠

- “学习工作台”包含导入、计划、执行、历史；“任务清单”也能改变任务状态，两个页面都能改变核心进度。
- “复盘”页面和托盘“生成今日复盘”都能触发复盘，但托盘结果不会进入当前 UI 状态。
- Prompt profile 在工作台用于导入和计划生成，同时设置页也可编辑；用户不容易理解“选择提示词”和“编辑提示词”的区别。
- 今日历史显示“版本”，数据库也有 `plan_versions`，但 UI 使用 daily plan 列表做历史，容易和真正 plan version 混淆。

### 4.2 冲突功能

- 托盘“开始当前学习块”不会开始学习块，只打开窗口和发通知。
- 托盘通知仍提示“打开今日计划”，但当前页面名称是“学习工作台”。
- Review 页面写着“人工确认”，但没有确认入口。
- AGENTS 中要求监控需用户明确启用，但当前开始 session 后自动启动 focus monitor，UI 没有独立的监控开关或暂停入口。
- AGENTS 中强调 AI 不应覆盖确认计划；当前生成计划确实只生成草稿，但确认多个计划时没有单一正式计划规则。

### 4.3 入口过多或不一致

- 复盘有页面入口和托盘入口，但结果展示路径不一致。
- 学习执行可从工作台块按钮进入，也有托盘“开始当前学习块”入口，但后者不执行真实动作。
- 设置里的 prompt 编辑入口太接近普通模型设置，容易把“运行配置”和“AI 行为配置”混成一类。

## 5. 用户可能不知道下一步做什么的位置

### 阻断级

- 首次启动缺 API Key 时，顶部只显示“需要 API Key”，没有按钮跳转到设置。
- 工作台没有任务时，“生成今日草稿”被禁用，但用户不一定知道必须先粘贴计划并导入。
- 导入失败时只通过 notice 显示错误，不保留可见的失败导入记录或重试入口。
- 计划生成失败时显示简化错误，但没有指向可操作的恢复路径。

### 操作理解级

- 导入成功后，文本框被清空，但用户看不到刚才导入创建了哪些任务。
- 生成草稿后，用户可能不知道应该先审查计划块还是直接确认。
- “今日历史”的版本按钮没有说明不同版本之间的差异。
- 学习块的“开始”是图标按钮，没有文字，第一次使用不明显。
- “跳过”按钮会触发浏览器 prompt，用户不知道跳过后会对任务状态造成什么影响。
- 完成学习块后，用户不知道任务是否被整体完成，还是只完成了一个切片。
- 复盘生成后，用户不知道下一步应该确认、保存、应用建议，还是返回工作台。

## 6. 页面之间数据和状态不一致的位置

### 6.1 工作台和任务清单

- 生成计划不会把任务状态改成 planned，因此任务清单里任务仍可能显示待安排。
- 完成一个学习块会把关联任务直接改成 done，即使任务原本被拆成多个块。
- 跳过学习块只改 block 状态并写 skip log，不改 task 状态。
- 任务清单手动改状态后，今天已有计划块不会同步更新或提示冲突。

### 6.2 工作台和复盘

- 复盘读取 `getDaySnapshot(date)`，其中 plans 按日期过滤，但 sessions 没有按日期过滤。
- 复盘生成结果仅保存在当前 React state，切换页面或刷新后不显示历史复盘。
- 托盘生成复盘会写入数据库，但不会更新复盘页面的 `review` state。

### 6.3 工作台和托盘

- 托盘“开始当前学习块”不读取当前计划块，也不调用 session start。
- 托盘“生成今日复盘”没有成功反馈和结果入口。
- 托盘文案仍使用“今日计划”，与当前“学习工作台”页面命名不一致。

### 6.4 UI 状态和持久化状态

- `activeSession` 只存在 renderer state 中，刷新或重启后不会恢复当前 active session。
- 完成 session 后 `activeSession` 被设置为 completed session，而不是清空，当前专注块仍可能显示。
- `FocusMonitor` 在主进程中运行，但 UI 没有展示监控是否正在运行，也没有暂停监控入口。

## 7. 当前 UI 中存在的信息层级问题

### 学习工作台

- 左栏同时包含导入和生成两个主动作，首次使用时“导入到今日上下文”和“生成今日草稿”的先后关系不够强。
- 中栏计划列表是视觉中心，但没有突出“当前推荐开始的学习块”。
- 右栏同时展示完成度、历史、专注块，其中“当前专注块”在执行阶段应该更强，但目前只是普通侧栏区域。
- 今日历史是重要状态，但被放在完成度下面，且缺少差异信息。
- 空状态占位说明正确，但没有把用户直接带到下一步操作。

### 任务清单

- 任务列表只突出状态和难度，不突出目标、估时、验收标准。
- 任务状态下拉框很强，但其影响范围不清楚，可能导致用户误改任务完成状态。

### 复盘

- 左侧“生成”是主按钮，但生成后没有“采纳建议”或“生成调整计划”的后续层级。
- 右侧“人工确认”看起来像真实模块，但没有控件，信息层级上高于实际能力。

### 设置

- 模型设置、API Key、块长度、Prompt 编辑混在一个页面。
- Prompt 编辑能力影响 AI 行为，风险比普通设置更高，但视觉上只是侧栏普通表单。
- 没有显示 API Key 保存成功时间、模型测试、连接测试或错误分类。

## 8. 当前主要按钮、次要按钮和危险操作是否合理

### 合理的部分

- “生成今日草稿”作为 primary button 合理，因为有任务后它是主要推进动作。
- “确认草稿”作为 primary button 合理，因为它把 AI 草稿变成正式计划。
- “保存设置”作为 primary button 合理。
- “保存新版本”作为 secondary button 合理，因为 prompt 修改是高级操作。
- “刷新”使用 icon button 合理。

### 不合理或风险较高的部分

- “导入到今日上下文”是首次流程的关键动作，但使用 secondary button，优先级偏低。
- 学习块“开始”只有图标，没有文字，主执行动作不够明确。
- “跳过”是会改变计划状态并写 skip log 的操作，但只是 text button，危险性和后果表达不足。
- 任务状态下拉可以直接把任务改成 done/skipped，没有确认，也没有说明会影响规划。
- “完成”按钮没有显示将会把关联 task 标记为 done。
- 复盘“生成”是 primary，但生成结果没有后续确认动作，主按钮后缺少闭环。
- 托盘“生成今日复盘”是隐藏入口，可能绕过页面中的用户理解和结果呈现。
- 关闭窗口隐藏到托盘是合理桌面行为，但没有首次提示，用户可能误以为应用已经退出。

## 9. 真实可用功能与视觉占位

### 真实可用

- Electron 主窗口、托盘、窗口隐藏和退出。
- SQLite 数据库 bootstrap 和本地持久化。
- DeepSeek API Key 加密保存。
- DeepSeek base URL、model、默认学习块分钟数保存。
- Prompt profiles seed 和保存新版本。
- 粘贴计划创建 raw import。
- AI 解析导入并创建 goals、tasks、dependencies。
- 任务列表读取和状态更新。
- AI 生成日计划草稿。
- AI 计划输出本地归一化。
- daily plans、daily plan blocks、plan versions 保存。
- 今日计划历史选择。
- 草稿计划确认。
- 学习 session start、pause、complete。
- 学习块 skip 和 skip log。
- 前台应用变化记录。
- AI 复盘生成和 ai review 保存。
- 基础错误 notice 显示。

### 部分可用

- 当前专注块：能开始和完成，但没有计时器、恢复 active session、监控状态展示。
- Focus monitor：能记录前台窗口变化，但没有 away/return 聚合、focusScore 计算、暂停入口、排除规则。
- 复盘：能生成并展示本次结果，但不能查看历史、确认建议、生成调整计划。
- 今日历史：能切换 daily plan，但不能比较差异，也不是真正的 plan_versions UI。
- 设置：能保存部分配置，但没有 autoLaunch UI、学习时间窗 UI、连接测试。
- 托盘复盘：能调用服务，但成功结果不可见。

### 视觉占位或语义占位

- 复盘页“人工确认”区域。
- 托盘“开始当前学习块”。
- 完成度指标只是完成块数，不是真正学习质量或任务完成度。
- 专注度字段存在，但当前没有实际评分计算。
- `evaluationAgentOutputSchema` 存在，但没有 evaluator-agent UI 和流程。
- `plan_versions` 有写入，但没有版本详情、diff 或回滚入口。
- goals 和 task dependencies 有保存，但 UI 不展示。
- RAG、知识库、长期记忆尚未实现。

## 10. 按严重程度分组的问题

### 10.1 阻断核心流程

1. **复盘没有形成计划调整闭环。**
   当前复盘只生成文字和下一步动作，没有确认、拒绝、应用建议、生成重排草稿的流程。核心 loop 在“reflection -> plan adjustment”处断开。

2. **完成一个学习块会直接完成整个任务。**
   如果一个任务被拆成多个块，完成任意一个块会把 task 标记为 done，破坏任务进度真实性。

3. **复盘 session 数据没有按日期过滤。**
   `getDaySnapshot(date)` 只按日期筛选 plans，但 sessions 使用所有历史 session，复盘可能基于错误数据。

4. **确认计划没有单一正式计划规则。**
   `confirmPlan` 只把指定 plan 设为 confirmed，不归档同日旧 confirmed 计划，也不处理多个草稿之间的关系。

5. **active session 状态不可恢复且完成后不清空。**
   当前专注块依赖 renderer state；刷新或重启后丢失，完成后仍可能显示 completed session。

### 10.2 影响理解和操作

6. **首次使用缺少明确引导。**
   缺 API Key、无任务、无计划时，用户需要自己推断去设置、导入、生成的顺序。

7. **导入成功后没有展示解析结果。**
   用户看不到 AI 创建了哪些任务、估时和验收标准，也无法在生成计划前确认导入质量。

8. **工作台缺少“当前应该开始哪个块”的推荐。**
   每个块都有开始按钮，系统没有突出当前块、下一个块或最优开始点。

9. **复盘结果没有历史入口。**
   生成结果在 React state 中显示，刷新或切页后无法从 UI 找回。

10. **状态变更后果不透明。**
   跳过、完成、任务状态下拉都会改变本地数据，但 UI 没有解释影响范围。

### 10.3 视觉一致性问题

11. **主操作层级不稳定。**
   首次流程中导入是关键动作却是 secondary；执行流程中开始是 icon-only；确认草稿是 primary，三者层级不一致。

12. **复盘右栏看起来像功能模块但实际不可操作。**
   “人工确认”视觉上占了侧栏空间，但没有交互，容易制造期待落差。

13. **今日历史的“版本”标签不准确。**
   UI 使用 plans 数组倒序显示“版本 N”，但这不是 `plan_versions.version`，容易误导。

14. **设置页混合普通配置和高风险 prompt 编辑。**
   Prompt 编辑对 AI 行为影响很大，但视觉上和普通设置并列。

15. **托盘和页面命名不一致。**
   托盘仍提示“今日计划”，主页面已经叫“学习工作台”。

### 10.4 可以延后处理

16. **任务清单缺少筛选、搜索和编辑。**

17. **目标、依赖、导入历史没有 UI。**

18. **学习时间窗和开机自启有后端能力但无 UI。**

19. **前台监控缺少排除规则和记录查看。**

20. **UI 自动化验证流程尚未稳定。**

## 11. 当前最需要优先修复的十个问题

1. 修复 `completeSession`：完成 block 不应默认完成整个 task，至少需要区分任务完成和块完成。
2. 修复 `getDaySnapshot(date)`：sessions 必须按日期过滤，复盘不能混入历史数据。
3. 明确同一天只能有一个 active confirmed plan，确认新计划时归档或降级旧计划。
4. 修复 active session 生命周期：完成后清空 renderer activeSession，并能从数据库恢复 active session。
5. 给首次使用增加强引导：缺 API Key -> 设置；无任务 -> 导入；有任务无计划 -> 生成草稿；有草稿 -> 确认。
6. 导入成功后展示解析出的任务摘要，并允许用户先检查再生成计划。
7. 复盘增加后续动作：保存/查看历史、生成调整建议、确认或拒绝，不要停在文字输出。
8. 托盘入口对齐真实能力：要么真正开始当前块，要么改名为“打开学习工作台”。
9. 让工作台突出“当前推荐学习块”，把开始学习变成明确主操作。
10. 明确危险和后果操作：跳过、完成、任务状态改为 done/skipped 前需要更清晰的提示或确认。

## 12. 审计结论

当前产品最值得保留的是本地优先架构、AI 输出校验、草稿确认边界、计划历史雏形和学习工作台方向。最需要马上修的不是视觉细节，而是数据状态和用户流程语义：任务完成、计划确认、复盘日期、active session、导入后确认和下一步引导。

建议下一阶段不要继续堆页面或加新 AI agent，先把“导入 -> 检查任务 -> 生成草稿 -> 确认计划 -> 开始当前块 -> 完成块 -> 复盘 -> 生成调整建议”的单线流程做稳。
